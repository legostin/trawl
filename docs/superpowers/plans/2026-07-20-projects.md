# http-catch — Проекты (scoped-захват, свои правила и хосты)

## Context

Пользователь редко отлаживает сразу все домены — обычно работает над конкретным сервисом. Нужен режим **проектов**: у каждого проекта свои правила, свой список отслеживаемых хостов (include) и исключаемых (exclude). Когда проект активен, запросы к «чужим» хостам **вообще не записываются** (экономия памяти), и работают только правила этого проекта. Хост можно добавить в проект прямо из истории запросов.

## Решения (согласованы)

- **Один активный проект** за раз. Проект выключен = текущее поведение (пишем всё, работают глобальные правила).
- При активном проекте — **только его правила** (глобальные, без проекта, действуют лишь когда проект выключен).
- **include/exclude — хост-глобы.** Голый хост матчит сам домен и его поддомены (`example.com` → `api.example.com`); можно и `*.example.com`.
- Захват: активный проект пишет только `include && !exclude`; нетрекаемые запросы **проксируются, но не сохраняются и не эмитятся**, правила к ним не применяются.
- «Добавить хост из истории» кладёт хост в include активного проекта.

## Модель

**Rust:**
- `projects.rs` — `Project { id, name, include_hosts: Vec<String>, exclude_hosts: Vec<String> }`; чистые `host_matches(entry, host)` (exact | subdomain | glob) и `project_tracks(project, host) = include.any && !exclude.any`; персистенция `projects.json` (список + `active_id: Option<String>`).
- `rules.rs` — в `Rule` добавляется `project_id: Option<String>` (None = глобальное). Матчинг правил в прокси фильтруется по активному проекту.
- Общие ячейки в `AppState`/прокси: `Arc<RwLock<Option<Project>>>` (активный проект, резолвнутые хосты) — чтобы хендлер знал, что трекать.

**Frontend:**
- `projects.ts` — стор: `projects`, `activeProjectId`, CRUD, `setActive`, `addHostToProject`.
- `Project`/`Rule` TS-типы (`projectId`).

## Задачи (TDD, чекпоинты, частые коммиты)

**1. Бэкенд: модель проектов + персистенция + `Rule.project_id`.**
`projects.rs`: структура, `host_matches`, `project_tracks`, load/save `projects.json` (+ active id). В `rules.rs` — поле `project_id`. Команды: `list_projects`, `save_project`, `delete_project`, `set_active_project`, `get_active_project`. Тесты: `host_matches` (точный/поддомен/glob), `project_tracks` (include/exclude), round-trip персистенции. *Проверка:* `cargo test`.

**2. Бэкенд: фильтр захвата + scoping правил в прокси.**
`CaptureHandler` получает `Arc<RwLock<Option<Project>>>` (активный) + активный `project_id`. В `handle_request`: если проект активен и `!project_tracks(host)` → вернуть запрос как есть, **без flow, без эмита, без правил**. Иначе — захват как сейчас; матчинг правил фильтруется: активный проект → `project_id == active`; выключен → `project_id == None`. `set_active_project` обновляет общую ячейку вживую. Wire в `start()`/`AppState`/`commands`. Тесты: нетрекаемый хост не сохраняется; трекаемый сохраняется + проектное правило применяется; глобальное правило игнорится при активном проекте. *Проверка:* `cargo test`.

**3. Фронтенд: стор проектов + селектор в TopBar.**
`projects.ts` (+ обёртки команд). В `TopBar` — дропдаун активного проекта: «Все домены (off)» / список проектов / «Управление…». Смена активного → `set_active_project` + перезагрузка списка потоков (`get_flows`). Имя активного проекта — в `StatusBar`. *Проверка:* `tsc`+`build`; вживую переключение.

**4. Фронтенд: редактор проектов.**
Панель/модалка: имя, список include-хостов (добавить/удалить), список exclude-хостов, создать/удалить проект. Открывается из дропдауна TopBar. *Проверка:* создание/правка проекта сохраняется и переживает перезапуск.

**5. Фронтенд: правила по проекту + «добавить хост из истории».**
`RulesView` показывает правила активного проекта (`projectId == active`) либо глобальные (когда off); новые правила (в т.ч. из истории) получают `projectId` активного проекта. В `FlowDetail`/строке трафика — действие «Добавить хост в проект» (кладёт `flow.url.host` в include активного проекта; если проектов нет — подсказать создать). *Проверка:* правило видно только в своём проекте; хост из истории попадает в include.

**6. Полировка + проверка.**
Индикатор активного проекта и «трекается N хостов» в StatusBar; пустые состояния («нет проектов»); живой прогон в приложении: активировать проект с одним хостом → трафик к другим хостам не появляется в списке, к трекаемому — появляется и ловится проектным правилом.

## Verification

- **Rust:** `cargo test` — host-match/tracks/персистенция; интеграция прокси (нетрекаемый хост не пишется; трекаемый пишется + проектное правило; глобальное игнорится при активном проекте).
- **Frontend:** `pnpm test`, `tsc`, `build`.
- **Живой прогон:** создать проект `example.com` (include), активировать → `curl` к `example.com` появляется, к другому хосту — нет (память не растёт); проектное правило применяется только в проекте; «добавить хост из истории» пополняет include; всё переживает перезапуск.

## Риски

- **Смена активного проекта на лету** — обновлять общую ячейку под RwLock; не держать guard через await.
- **Обратная совместимость правил** — существующие правила без `project_id` = глобальные (работают при выключенном проекте).
- **Матчинг поддоменов** — `example.com` матчит `*.example.com`; следить, чтобы `exclude` имел приоритет над `include`.
