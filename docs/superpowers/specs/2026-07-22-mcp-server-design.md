# MCP-сервер — дизайн

**Дата:** 2026-07-22
**Статус:** утверждён к реализации

## Что это

Кор-модуль: встроенный MCP-сервер (Model Context Protocol), через который
AI-агенты (Claude Code, Cursor и т.п.) получают доступ к Trawl — трафику,
правилам, проектам, брейкпоинтам — и могут ими управлять. Плагины могут
добавлять в сервер собственные тулы.

## Решения (зафиксированы при брейнсторминге)

| Вопрос | Решение |
|---|---|
| Транспорт | Streamable HTTP на `127.0.0.1:<port>` (дефолт 9910), внутри Rust-бэкенда |
| SDK | официальный `rmcp` (фича streamable-http-server, тянет axum) |
| Включение | тоггл в настройках UI, по умолчанию включён; конфиг в `mcp.json` (app data dir) |
| Аутентификация | обязательный `Authorization: Bearer <token>`; токен генерится при первом старте (32 случайных байта, hex), кнопка перегенерации в UI |
| Плагинные тулы | регистрация из JS через `__TRAWL__.mcp`, вызов через мост Rust ↔ webview |
| Брейкпоинты | агент сам разруливает пойманные запросы: `list_paused` + `resolve_breakpoint` (поллинг, без push-нотификаций в MVP) |

## Архитектура

Новый модуль `src-tauri/src/mcp.rs` (+ при необходимости `mcp_tools.rs` для
описаний тулов). Реализуем `rmcp::ServerHandler` вручную (не через макросы):

- `list_tools` — кор-тулы + плагинные из динамического реестра;
- `call_tool` — диспатч либо в кор-логику, либо в плагинный мост.

Сервер получает клон `AppHandle` и достаёт `AppState` (store, rules,
breakpoints, pending_breakpoints, db) так же, как Tauri-команды. Общую
бизнес-логику команд выносим в функции, которые зовут и Tauri-команды, и
MCP-тулы — один источник правды, единая валидация (конфликты правил и
брейкпоинтов и т.д.).

### Жизненный цикл

- Старт приложения → если `enabled` в `mcp.json` — поднимаем сервер.
- Тоггл в UI поднимает/гасит сервер на лету.
- Конфиг `mcp.json`: `{ "enabled": bool, "port": u16, "token": string }`.
- Порт занят → сервер не поднимается; статус и текст ошибки видны в блоке
  MCP в настройках, там же можно сменить порт.

### Безопасность

- Биндинг только на loopback.
- Запрос без токена или с неверным → HTTP 401.
- В UI — блок «MCP»: статус, порт, токен, перегенерация, готовая команда
  для копирования:
  `claude mcp add --transport http trawl http://127.0.0.1:9910/mcp --header "Authorization: Bearer <token>"`.

## Кор-тулы

Имена — snake_case, ответы — JSON.

### Статус

- `get_status` — запущен ли прокси, адрес:порт, активный проект,
  intercept вкл/выкл, число флоу в памяти и в базе.

### Трафик

- `query_flows(filter, limit, offset)` — строки SQLite-истории (метод, host,
  path, статус, размеры, тайминги, `applied_rules`, projectId) **без тел**.
  Фильтр — существующий `FlowQuery` (подстрока, method, statusClass, host,
  projectId, диапазон времени).
- `get_flow(id, max_body_bytes=50000)` — флоу целиком: заголовки, тела,
  применённые правила, тайминги. Текстовые тела режутся по `max_body_bytes`
  с флагом `truncated`; бинарные — только размер и content-type.
- `flow_count(filter)` — количество по фильтру.
- `aggregate_flows(filter, group_by)` — агрегаты по хостам / статус-классам /
  времени / длительности.

### Правила

- `list_rules(project_id?)` — паттерн, фаза, enabled, скрипт.
- `save_rule(rule)` — создать/обновить (без id → генерим); валидация как в
  UI: конфликт двух включённых правил на паттерн+фазу → ошибка с объяснением.
- `delete_rule(id)`.
- `get_scripting_reference` — справка по API скриптов правил (`ctx`, фазы,
  env, `breakpoint()`) + исходник общей библиотеки, чтобы агент мог грамотно
  писать скрипты.

### Проекты

- `list_projects`, `save_project(project)`, `delete_project(id)`,
  `set_active_project(id?)` — включая include/exclude hosts и env-переменные.

### Брейкпоинты

- `list_breakpoints`, `save_breakpoint(bp)`, `delete_breakpoint(id)` — CRUD
  определений, валидация конфликтов как в UI.
- `list_paused` — стоящие на паузе флоу (id, фаза, метод, url, заголовки,
  тело), чтобы агент решил, что менять.
- `resolve_breakpoint(flow_id, phase, action, edits)` — `execute` (пропустить
  с правками), `respond` (ответить самому), `abort`. Маппится на существующий
  `Resolution`.

### Отправка

- `send_request(method, url, headers, body, via_proxy?)` — как
  Composer/httpsend; ответ возвращается агенту с тем же лимитом на тело.

## Плагинные тулы

Новый раздел host API:

```ts
__TRAWL__.mcp = {
  registerTool({ name, description, inputSchema, handler, timeoutMs? }),
  unregisterTool(name),
}
```

- `registerTool`: фронт держит `handler` в карте, метаданные (имя, описание,
  JSON Schema входа) отдаёт в Rust командой `mcp_register_tool(pluginId, …)`.
  Rust кладёт их в динамический реестр и шлёт клиентам
  `notifications/tools/list_changed`.
- Итоговое имя тула — `<pluginId>_<name>` (например `http-client_send`) —
  без коллизий между плагинами и кор-тулами.
- Вызов: Rust генерит `call_id`, эмитит Tauri-событие `mcp:tool-call`
  `{callId, tool, args}` в webview, ждёт oneshot. Фронт выполняет handler и
  возвращает результат командой `mcp_tool_result(callId, result | error)`.
- Таймаут 60 с по умолчанию (переопределяется `timeoutMs`); по истечении —
  ошибка тула.
- Жизненный цикл: перед (пере)инъекцией бандла плагина хост чистит его тулы
  из реестра; disable — тоже чистит. Плагин при загрузке регистрируется
  заново.

## Обработка ошибок

- Ошибки бизнес-логики (конфликт правил, «no pending breakpoint», флоу не
  найден) → MCP-результат с `isError: true` и человекочитаемым сообщением.
- Неверный/отсутствующий токен → HTTP 401. Кривой JSON-RPC → стандартные
  ошибки протокола (rmcp).
- Порт занят → сервер не стартует, ошибка видна в настройках.
- Упавший/затаймаутивший JS-handler плагина → ошибка тула с именем плагина
  в тексте.

## Тестирование

- **Rust:** юнит-тесты диспатча тулов на общих функциях ядра (валидация
  правил/брейкпоинтов, `query_flows` по временной SQLite, маппинг
  `resolve_breakpoint`); аутентификация (401 без токена, 200 с ним);
  `tools/list` отражает динамическую регистрацию/удаление.
- **TS (vitest):** мост — регистрация/очистка при перезагрузке плагина,
  диспатч события в handler, возврат результата и ошибки.
- **Ручная проверка:** подключить Claude Code к живому приложению: запрос
  трафика, создание правила, разруливание брейкпоинта.

## Вне объёма (MVP)

- stdio-транспорт / отдельный бинарник-шим.
- Push-нотификации агенту о пойманном брейкпоинте (server-initiated события
  плохо поддержаны клиентами) — кандидат на потом.
- Start/stop прокси через MCP.
- MCP resources/prompts — только tools.
