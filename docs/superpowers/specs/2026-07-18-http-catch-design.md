# http-catch — дизайн

**Дата:** 2026-07-18
**Статус:** утверждён к реализации

## Что это

Десктоп-приложение — аналог Charles Proxy с удобным интерфейсом, собранное
на Tauri. MITM HTTP(S)-прокси: перехватывает трафик, расшифровывает HTTPS
через собственный CA и показывает запросы/ответы в UI.

**Главный сценарий MVP:** отладка **мобильного трафика** — телефон ходит через
наш прокси по Wi-Fi, HTTPS расшифровывается нашим CA-сертификатом,
разработчик видит и анализирует запросы приложения на Mac.

## Решения (зафиксированы при брейнсторминге)

| Вопрос | Решение |
|---|---|
| Целевая платформа MVP | только macOS |
| Главный сценарий | мобильный трафик через Wi-Fi-прокси + CA для HTTPS |
| Прокси-ядро | своё, на Rust (`hudsucker` / `hyper` / `rustls` / `rcgen`) внутри Tauri-бэкенда |
| UI-стек | React + TypeScript |
| Фичи MVP | просмотр трафика, фильтр/поиск, повтор/редакт запроса, breakpoints, экспорт (HAR/curl) |
| Хранение | гибрид: в памяти во время работы + ручной save/load сессии в файл |

### Почему своё ядро на Rust, а не mitmproxy

- Tauri-бэкенд уже на Rust — прокси живёт в том же процессе, без внешнего
  процесса и IPC-моста.
- Один бинарник без внешних рантаймов; mitmproxy тащит Python, что тяжело
  для поставки/подписи/нотаризации на macOS.
- `hudsucker` даёт готовый MITM-прокси с генерацией leaf-сертификатов на лету
  под свой CA; хендлер подключается к каждому запросу/ответу.
- Прямой путь событий: прокси-хендлер → Tauri event → фронтенд.

Компромисс: граничные случаи (WebSocket, chunked, gzip/brotli, HTTP/2) —
на нас, но `hudsucker`/`hyper` покрывают большую часть; для MVP достаточно.

## Общая архитектура

Один Tauri-процесс, две половины — React-фронтенд в WebView и Rust-бэкенд.

```
┌─────────────────────── Tauri App (один процесс) ───────────────────────┐
│   React + TS (WebView)              Rust backend                        │
│  ┌───────────────────┐            ┌──────────────────────────────┐      │
│  │ Traffic list       │◀── events─│ Proxy engine (hudsucker)     │      │
│  │ Request/Resp detail│            │  - TCP listen :8888          │      │
│  │ Filter / search    │            │  - MITM: leaf-cert под CA    │      │
│  │ Repeat / Compose   │── cmd ────▶│  - hook: on_request/on_resp  │      │
│  │ Breakpoints panel  │            ├──────────────────────────────┤      │
│  │ Setup / cert helper│            │ Session store (in-memory)    │      │
│  └───────────────────┘            │ CA manager (rcgen)           │      │
│                                    │ Exporter (HAR / curl)        │      │
│                                    └──────────────────────────────┘      │
└─────────────────────────────────────────────────────────────────────────┘
        ▲ Wi-Fi (телефон → HTTP proxy 192.168.x.x:8888)
```

## Компоненты бэкенда (Rust)

Каждый — отдельный модуль с одной ответственностью и чётким интерфейсом.

- **`proxy`** — оборачивает `hudsucker`. Слушает порт, при HTTPS подменяет
  сертификат под свой CA. На каждый запрос/ответ формирует `Flow` и шлёт
  событие во фронт. Точка внедрения breakpoints. Отдаёт спец-URL `/cert`
  (см. мобильный setup) до MITM.
- **`ca`** — генерирует и хранит корневой CA (`rcgen`) в app data dir
  (создаётся один раз при первом запуске). Отдаёт публичный `.pem`/`.crt`
  для установки на телефон.
- **`store`** — держит `Flow`-и в памяти (`Vec` + лимит по количеству,
  старое вытесняется — ring buffer). Отдаёт по id, чистит, сериализует
  для save/load.
- **`exporter`** — превращает `Flow` в HAR (сессия целиком) и в `curl`
  (один запрос).
- **`commands`** — Tauri-команды из UI: start/stop proxy, get flow, replay,
  set/resume/abort breakpoint, export, save/load session, get cert path,
  get proxy address.

## Компоненты фронтенда (React)

- **`TrafficList`** — виртуализированная таблица (метод, host, path, статус,
  размер, время). Живо обновляется по событиям.
- **`FlowDetail`** — вкладки Headers / Body (JSON-вьюер + подсветка) / Timing
  для запроса и ответа.
- **`FilterBar`** — фильтр по host/status/method + текстовый поиск.
- **`Composer`** — форма повтора: берёт flow, даёт отредактировать и
  отправить заново.
- **`BreakpointsPanel`** — список правил-паттернов; при пойманном потоке —
  модалка правки перед отправкой.
- **`SetupPanel`** — показывает IP:порт прокси, кнопку/QR для скачивания CA
  + пошаговую инструкцию для iOS/Android.

## Модель данных `Flow`

Общая для Rust и TS:

- `id`
- `timestamp`
- `method`
- `url` (scheme / host / port / path)
- `request { headers, body }`
- `response { status, headers, body }`
- `timings { sent, ttfb, done }`
- `state` — `pending | completed | error | paused`

Тела хранятся как байты + признак «текст/бинарь». В UI большие/бинарные тела
не рендерятся целиком — показываются по кнопке (лимит на авторендер).

## Мобильный сценарий — как телефон попадает в прокси

`SetupPanel` ведёт по шагам:

1. **Показать адрес.** Приложение определяет LAN-IP Mac и показывает
   `192.168.x.x:8888` крупно. Телефон и Mac — в одной Wi-Fi-сети.
2. **Настроить прокси на телефоне.** Инструкция: Wi-Fi → настройки сети →
   HTTP-прокси вручную → IP и порт.
3. **Установить CA.** На телефоне открыть `http://<ip>:8888/cert` (прокси
   отдаёт публичный CA по этому спец-URL) → скачать → установить и **вручную
   доверить** (iOS: Settings → General → About → Certificate Trust Settings;
   Android: install as CA). Панель показывает шаги + QR со ссылкой.
4. **Проверка.** Кнопка «Проверить» ждёт первый расшифрованный HTTPS-поток
   и подтверждает работу.

Спец-URL `/cert` обслуживается самим прокси до MITM — телефон качает
сертификат через тот же порт, ничего дополнительно поднимать не нужно.

## Поток breakpoints

- В `proxy`-хендлере перед отправкой запроса и перед отдачей ответа
  проверяются правила брейкпоинтов.
- При совпадении: `Flow.state = paused`, событие в UI, async-задача
  удерживается через `oneshot`-канал.
- UI показывает модалку правки. «Resume» (возможно, с изменённым
  содержимым) → команда шлёт данные обратно в канал → хендлер продолжает.
- «Abort» — поток закрывается с ошибкой. Есть таймаут-страховка, чтобы
  забытый брейкпоинт не завис навечно.

## Обработка ошибок

- **Порт занят** — проверяем при старте; при конфликте понятная ошибка +
  предложение сменить порт.
- **Апстрим недоступен / TLS-ошибка** — поток `error` с причиной, виден
  в списке (красным), приложение не падает.
- **CA не доверен на телефоне** — HTTPS-потоки рвутся; `SetupPanel`
  подсказывает причину.
- **Большие/бинарные тела** — не грузятся в UI целиком (лимит), webview
  не залипает.
- **Переполнение памяти** — ring-лимит в `store` вытесняет старые потоки;
  лимит настраивается.

## Тестирование

- **Rust unit:** `ca` (валидный CA и leaf-cert), `exporter` (HAR-схема,
  curl-строка), `store` (лимит/вытеснение), правила фильтра/брейкпоинтов.
- **Rust интеграционные:** прокси на случайном порту, реальный HTTP-запрос
  через него → проверка сборки `Flow`; HTTPS через тестовый клиент,
  доверяющий нашему CA.
- **Фронтенд:** компонентные тесты списка/фильтра/composer на моках событий.
- **Ручной smoke:** реальный телефон через прокси на live-сайте — в чеклист
  релиза, не в CI.

## Порядок реализации (фазы)

Каждая фаза — рабочее приложение, ценность растёт инкрементально.

1. **Каркас + просмотр HTTP:** Tauri-скелет, `proxy` (только HTTP),
   event-поток, `TrafficList` + `FlowDetail`.
2. **HTTPS MITM + setup:** `ca`, MITM в прокси, `/cert`-эндпоинт,
   `SetupPanel`.
3. **Фильтр/поиск** в списке.
4. **Повтор / Composer.**
5. **Breakpoints.**
6. **Экспорт (HAR/curl) + save/load сессии.**

## Вне рамок MVP (осознанно отложено)

- Windows / Linux.
- Rewrite-правила, map local/remote, throttling.
- Постоянное хранилище (SQLite) — пока только in-memory + ручной save/load.
- WebSocket-инспектор как отдельная вкладка (базовый проброс — да, богатый UI — нет).
